# CC Translate — 迭代路线图（Roadmap）

本文件记录对 CC Translate 的迭代建议，来自一次整体代码走查。按优先级分层，
并标注当前状态。计划实施顺序：**1 → 3 → 2 → 5 → 6 → 4**，其余作为 backlog。

状态图例：⬜ 待办 · 🔄 进行中 · ✅ 已完成 · ⏸ 暂缓

---

## 🔴 高优先级（体验 / 正确性）

### 1. 保护用户剪贴板 ✅
- **问题**：双击 Ctrl+C 触发翻译时，用户选区会覆盖剪贴板原有内容，且从不还原，
  破坏用户正在进行的复制粘贴工作流（这类工具最常见的痛点）。
- **方案**：触发翻译前先保存当前剪贴板内容，读完选区后延迟把原内容还原回去。
- **风险**：低。仅在触发路径增加一次保存/还原。

### 3. 用户数据迁移到 %APPDATA% ✅
- **问题**：`config.json` / `history.json` / `perf.log` 都写在仓库目录（`APP_DIR`）。
  虽已被 gitignore，但会导致：只读安装失败、`git clean` 误删历史、多副本互相打架。
- **方案**：改用 `%APPDATA%\CC Translate\` 存放用户数据；首次启动时若旧位置存在数据则自动迁移。
- **附带收益**：为后续 macOS 移植（`~/Library/Application Support/`）铺路。
- **风险**：中。需保证向后兼容与一次性迁移。

### 2. 关键路径异常日志化 ✅
- **问题**：全文大量 `except Exception: pass` 静默吞掉异常，任何 bug 都无迹可寻，排查困难。
- **方案**：保留兜底不崩溃，但在关键路径（翻译调用、流式、剪贴板、配置读写、UI 构建）
  把异常写入 `perf.log`（复用已有 `log_perf`）。
- **风险**：低。只增加日志，不改变控制流。

## 🟡 中优先级（功能增强）

### 5. 历史详情富文本渲染 ✅
- **问题**：历史窗口的详情区是纯文本（`_rich=False`），与主弹窗的富文本体验不统一。
- **方案**：历史详情复用主弹窗的 markdown-lite 渲染，样式统一。
- **风险**：低。复用现成渲染器。

### 6. 代码块语法高亮（Pygments）✅
- **问题**：目前代码块只是等宽 + 灰底，没有按 token 上色。
- **方案**：为 ``` fenced 代码块接入 Pygments，按 token 类型着色，代码解释模式更像 VSCode。
  仅在定稿帧做，不进流式热路径，保证翻译速度不受影响。
- **风险**：中。新增可选依赖（缺失时优雅降级为纯等宽）。

### 4. 弹窗内一键换向 / 重译 ✅
- **问题**：译完想换翻译方向（如强制中→英）得进设置改，不顺手。
- **方案**：结果弹窗加一个小控件，一键切换方向并对当前选区重新翻译。
- **风险**：低。复用现有翻译管线。

## 🟢 Backlog（长期 / 架构，暂缓）

### 7. 分类阈值优化 ⏸
- `classify_selection` 纯靠字符比例 `code_ratio`，边界情况（一句话夹一个 `foo()`、
  纯 JSON 数据）可能误判。可补充信号（换行密度、缩进、括号配对）。
- **建议**：先收集真实使用中的误判样本再调，避免过拟合。

### 8. 单文件拆分 ⏸
- `translator.pyw` 单文件约 2700+ 行，建议拆为
  `config.py` / `themes.py` / `richtext.py` / `claude.py` / `ui.py` / `tray.py`。
- 纯内部重构，对外行为零变化，提升可维护性与「给 AI 助手改」的友好度。
- **建议**：功能稳定后集中做一次。

### 9. 单元测试 ✅
- 纯函数（`classify_selection`、`code_ratio`、`iter_rich_segments`、`is_single_word`、
  `highlight_code`）最适合加单元测试，投入小、回归保护大。
- **已完成**：`tests/` 目录，37 个用例，覆盖分类/词典判定/markdown-lite 解析/
  流式安全/代码块高亮降级。用标准库 `unittest`，零新运行时依赖，对 app 功能无影响。
  跑法：`python -m unittest discover -s tests`（或 `pytest`）。

### 10. 打包分发 ⏸
- 用 PyInstaller 打成单个 `.exe`，普通用户双击即用，免装 Python 与依赖。

### 11. 自动检查 / 更新 ✅
- **需求**：本地程序能检查 GitHub 是否有新版本，有则更新并重启；支持夜间自动更新
  （设置里可开关）与手动更新。
- **方案**：app 本身是 `git clone` 部署，所以「更新 = `git pull --ff-only`」。
  - 检查：`git ls-remote origin master` 取远端 SHA 与本地 `HEAD` 比对（不落文件、不改本地）。
  - 更新：`git fetch` → `merge-base --is-ancestor` 确认可 fast-forward（本地分叉/领先则中止）
    → `git merge --ff-only`。
  - 安全护栏：更新后 `py_compile` + 跑单元测试，任一失败自动 `git reset --hard` 回滚并放弃重启。
  - 重启：启动一个分离的中转脚本（独立 `.ps1` 文件），轮询等当前进程退出（释放单例互斥锁）后再拉起新实例并在其秒退时重试，避免锁竞争；全过程写 `relaunch.log` 便于诊断。当前进程另有 `os._exit(0)` 兜底，确保非守护线程不会拖住旧进程、卡住重启。
    - ⚠️ 关键坑：中转脚本必须用 `CREATE_NO_WINDOW` **单独**创建，不能带 `DETACHED_PROCESS`。从无控制台的 `pythonw` 宿主派生时，`DETACHED_PROCESS`（无论是否与 `CREATE_NO_WINDOW` 叠加）会让无窗口的 PowerShell 助手根本不执行，导致重启静默失败；同时把子进程 stdio 重定向到 `DEVNULL`（`pythonw` 没有可继承的标准句柄）。
  - 无人值守安全：所有 git 调用禁用凭据弹窗（`GIT_TERMINAL_PROMPT=0`）+ 短超时，夜间检查不会卡住。
  - 优雅降级：非 git 部署 / git 缺失 / 网络失败均静默跳过，不影响正常使用。
- **配置**：`auto_update_enabled`（默认开，凌晨检查）、`auto_update_hour`（默认 3）。
- **入口**：设置「更新」区（版本号 + 夜间自动更新开关 + 检查更新按钮 + 发现新版本时才出现的「更新并重启」按钮）、托盘「检查更新」。手动「检查更新」只检查不改动，发现新版由用户点按钮决定；托盘「检查更新」会打开设置并触发同一处检查，两个入口收束到一处。夜间自动更新则在后台静默完成。
- **版本标识**：git 短 SHA + 提交日期（如 `9ef3615 · 2026-07-13`）。
